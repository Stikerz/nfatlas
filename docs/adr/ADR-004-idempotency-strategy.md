# ADR-004: Idempotency strategy (client-supplied keys, request-hash table)

**Status:** Approved
**Date:** 2026-06-29
**Approved by:** S1408661 (Engineering Lead) on 2026-07-02
**Reversibility:** Two-way door.

## Context

`PRD.md §3.5` requires ticket purchase to be **end-to-end idempotent** on a client-supplied request key — double-clicking "Buy 10 tickets" must produce exactly 10 tickets, never 20. `PRD.md §3.4` requires every Paystack payment intent to be idempotent. The ledger schema (ADR-003) gates duplicate journal posts on a unique `idempotency_key` index.

This ADR commits the mechanism that ties these together.

## Decision

### Client-supplied keys

Every mutation-causing endpoint accepts an `Idempotency-Key` request header (UUIDv4 generated by the client). For endpoints that mutate state without a key, the server generates one from `hash(user_id || request_body || endpoint || minute_bucket)` — this protects against the most common case (a retry of the same request shortly after the original) but is a fallback, not the primary mechanism.

### Server-side idempotency table

```sql
CREATE TABLE idempotency_records (
  key           TEXT PRIMARY KEY,            -- the Idempotency-Key value (or fallback hash)
  user_id       UUID,                         -- nullable for unauthenticated calls
  endpoint      TEXT NOT NULL,                -- "POST /api/v1/tickets" etc.
  request_hash  TEXT NOT NULL,                -- SHA-256 of the canonical request body
  response_code INT,                          -- HTTP status of the recorded response
  response_body JSONB,                        -- recorded response body
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  completed_at  TIMESTAMPTZ                   -- null while in flight
);

CREATE INDEX idempotency_records_created_idx ON idempotency_records (created_at);
```

### Processing

On every idempotent request:

1. **Lookup** by `key`.
2. **If found and `completed_at` is set:** verify `request_hash` matches; if it doesn't, return `409 Conflict` (the key was reused for a different request body). If it does, return the recorded `response_code` + `response_body` verbatim.
3. **If found and in flight** (`completed_at IS NULL` and `created_at` < 30s ago): return `409 Conflict` with `Retry-After: 1`. The client retries; eventually the original completes or its row ages out.
4. **If not found:** insert the row in a transaction with the actual business write. On success, record the response and set `completed_at`. On failure, the row rolls back and the next retry treats it as new.

### Scope

Endpoints that require idempotency keys:

- `POST /api/v1/payments/intents` (Payment module)
- `POST /api/v1/tickets/purchase` (Ticket module — covers paid + free-entry transcription submission)
- `POST /api/v1/withdrawals` (added when withdrawals ship; not V1)
- `POST /api/v1/draws/{id}/commit` (Draw Engine — operator action; idempotent so a network blip doesn't produce two commits)
- `POST /api/v1/draws/{id}/reveal` (same reasoning)

Read endpoints and admin queries do not require idempotency keys.

### Retention

Idempotency records are retained for **30 days** then purged by a scheduled job. 30 days covers all realistic retry windows including Paystack webhook re-delivery.

## Alternatives considered

- **Application-level deduplication keyed on `(user_id, endpoint, request_hash)` only.** Lost: relies on the request body being byte-identical on retry, which fails when clients add timestamps or correlation IDs to bodies. Client-supplied keys are explicit about intent.
- **Database-level uniqueness constraints alone.** Considered for the ticket purchase path (the ledger `idempotency_key` unique index would catch duplicates). Lost as a sole mechanism: gives a duplicate-key error to the client rather than the original response — leaks implementation detail and produces poor UX on retry.
- **Distributed lock (Redis SETNX).** Lost: introduces a second stateful infra dependency for the lock; Postgres handles it fine with the in-flight check.
- **Per-endpoint custom dedup logic.** Lost: scales poorly; a single shared mechanism is reviewable and testable.

## Consequences

**Positive:**
- Clients control retry semantics. Mobile app's offline queue (per Frontend Engineer agent's offline-first policy) can resubmit confidently after reconnect.
- Webhook re-delivery from Paystack is automatically deduplicated.
- The idempotency record + the ledger's unique index on `idempotency_key` give defense in depth: the application catches the duplicate; the database catches the application bug.

**Negative:**
- One extra DB write per mutation. Negligible at V1 volumes.
- 30-day retention means the table accumulates ~30 × daily mutation count rows. Manageable; the scheduled purge keeps it bounded.
- Clients must generate a UUID per intended mutation. SDK helpers (in `apps/mobile/` and `apps/admin/`) wrap this so application code rarely sees it.

**Invariants:**
- Every mutation endpoint listed in §Scope rejects requests without an `Idempotency-Key` header (or generates one server-side per the fallback rule, recorded as such).
- Reuse of a key with a different request body returns 409, not a silent re-execution.
- A QA test in `e2e/` covers the double-click ticket-purchase scenario (critical journey #8 in `delivery-framework.md §8`).
