## 0. Why this exists

Founder decision on 2026-07-02: **defer managed-platform commitment**; build a working local demo (V0.5) first. Primary audience: investors. Secondary: self-validation + counsel demonstration of the prize-competition mechanics.

This is a lean-startup move: prove the flows and unit-economics story in software before committing to cloud vendor lock-in. It also produces an artefact that unblocks fundraising conversations without waiting for full V1 launch.

**V0.5 is not throwaway.** It is a scoped subset of Phase 3 module implementation. Extending V0.5 → V1 is additive (real vendors, missing flows, hardening), not a rewrite.

---

## 1. Audience and success criteria

### Primary audience: investors

Success = a founder can screen-share the demo on a Zoom call and walk through the flagship flow in ≤ 5 minutes without a crash, a stumble, or a "this isn't wired up yet." Audience walks away with three takeaways:

1. **The mechanic works.** Prize competition (free entry route + skill question) is real, not slideware.
2. **The trust story is provable.** The audit log page shows the hash chain; the verifier link works; the draw output is reproducible from published inputs.
3. **The team can execute.** The product exists as running software, not diagrams.

### Secondary audience: self-validation

Success = the founder personally goes through every V0.5 flow at least 20 times before demoing to anyone else, catches edge cases, and updates the demo based on real friction discovered.

### Secondary audience: counsel

Success = when counsel (per `legal/counsel-engagement-brief.md`) meets Atlas, they can be shown the mechanic in operation. This accelerates their Q1 work — reading a description of "free entry route + skill question" is slower than seeing them.

### Anti-success criteria (things V0.5 is NOT for)

- ❌ Real users. Not accepting real payment or real KYC data.
- ❌ Load testing. Single-user demo runs only.
- ❌ Regulatory launch readiness.
- ❌ A production-quality codebase. Some short-term shortcuts are acceptable (documented, ticketed for V1 removal).

## 2. Scope — flagship flow (what V0.5 must do)

Numbered so scope conversations reference numbered items, not vibes.

### Consumer flow (Flutter app on founder's laptop)

1. **Register** — email + phone → OTP (from Mailhog) → password → logged in.
2. **Browse the active draw** — one draw configured in seed data. Prize (cash ₦2M or a car photo), close time, current entry count (paid + free split visible).
3. **Free-entry disclosure** — draw page shows the free-route mechanic prominently.
4. **Buy paid ticket** — click "Enter" → skill question (multiple choice, rotated from seed pool) → correct answer → Paystack sandbox checkout → ticket appears in "My tickets."
5. **View ticket** — ticket number, purchase timestamp, entry source ("paid").
6. **Draw completes** — user sees notification (in-app + Mailhog email) whether they won.
7. **Winner claim start** — if won, tap "Claim" → prize claim form → submit → status shows "claim received."

### Operator flow (Next.js admin on founder's laptop)

8. **Login** — operator credential from seed.
9. **Create draw** — form: prize description, ticket price, entries cap, close time, draw time, skill question set. Publish commit hash on save.
10. **Transcribe free entry** — form: enter user identifier + entry slip reference → creates a free-route ticket in the same pool.
11. **Close draw** — button. Sales stop. `tickets_hash` computed and displayed.
12. **Reveal draw** — button (at scheduled reveal time; can be manually triggered in demo). Public entropy fetched (Bitcoin block + drand), server seed revealed, winner selected deterministically, published.
13. **View audit log** — filterable table of every recorded event for the draw, chain-verified inline.

### Public verification (the wow moment)

14. **Proof page** — public URL (no login) showing: commitment hash, ticket list hash, revealed server seed, entropy inputs, algorithm reference, computed winner. Copy-paste-runnable verifier script snippet.

### Trust-story pages (static, seed-data-backed)

15. **How it works** — one page explaining prize competition (not lottery), free entry route, skill question, provably fair draw.
16. **Responsible play** — self-exclusion mentioned, permanent commitment stated.

## 3. Scope — explicitly out for V0.5

Everything below is designed and reserved for Phase 3 full V1, not V0.5:

- ❌ **Real KYC vendor integration** — V0.5 uses `KYCProvider` interface with a "manual approve in admin" stub. Interface (per ADR-007) is real; concrete adapter is stubbed.
- ❌ **Real payment webhook** — Paystack sandbox handles the payment; V0.5 receives its webhook and processes it. No real money. No real settlement reconciliation.
- ❌ **Real WhatsApp Business API** — winner notification goes to Mailhog email only; WhatsApp shown as a "coming soon" UI element in V0.5.
- ❌ **Real self-exclusion enforcement** — schema and tables present, BVN pepper stubbed with a fixed value; enforcement code exists but is not exercised in demo flows.
- ❌ **Full refund flow** — happy-path claim only. Refund flow demoed as "admin can initiate; state machine not built out."
- ❌ **Multiple concurrent draws** — one active draw at a time.
- ❌ **Multi-user concurrency** — single-user demo path.
- ❌ **Prize fulfilment state machine** — shows `winner_selected → winner_contacted` transitions; other states out.
- ❌ **RBAC beyond superadmin + user roles** — demo has one operator and one user role. Full 5-role RBAC (ADR-009) deferred.
- ❌ **NDPA compliance surface** — no DPO appointment, no NDPC registration, no DPIA. Not shippable to real users.
- ❌ **Managed-platform deploy** — local Docker Compose only.
- ❌ **Load / performance targets** — no k6, no perf gates.
- ❌ **Observability stack** — Sentry not wired; local logs to stdout only.
- ❌ **CI beyond lint + typecheck + unit tests** — deploy pipelines deferred to Phase 5.

Everything cut here has a **V1 story that reintroduces it.** No feature is silently dropped.

### Non-negotiables preserved even at V0.5

These are the design invariants that must hold in V0.5 (breaking them undermines the trust story the demo exists to tell):

1. **Double-entry ledger** — no `balance` column anywhere; all balances derived from journal (ADR-003 invariant).
2. **Audit log** — append-only + hash-chained; `INSERT`-only application role in local DB (ADR-005 invariant).
3. **Commit-reveal** — server seed committed at draw creation; revealed only after close; deterministic algorithm produces same winner on replay (ADR-006 invariant).
4. **Idempotency** — end-to-end on ticket purchase (ADR-004 invariant).
5. **Modular monolith module boundaries** — no cross-module direct DB access (grep invariant from PRD §4).

These invariants exist to prevent shortcuts that would compromise the trust story. Demo shortcuts that cross them are demo-quality-killers.

## 5. Sequence — ~7 weeks

### Week 1–2: Design pass (🎨 Sally leads)

**Deliverables:**

- Wireframes for all 16 flagship flow steps (§2).
- Mini design system: colour tokens, typography scale, ~15 core components (button, input, ticket card, draw card, timer, table row, banner, skill-question card, audit-log-row, verification-badge, empty state, error state, modal, toast, nav bar).
- Copy for prize-competition explainer page, responsible-play page, free-entry-route disclosure.
- Two Figma files: consumer (Flutter) + admin (Next.js).
- Accessibility-first from the start (WCAG 2.2 AA, per V1 target).

**Exit gate:** founder review of the flagship-flow wireframes; sign-off before Amelia starts building against them.

### Week 3–7: Build (💻 Amelia leads, 🛡️ Tobi supports on infra)

**Week 3:** repo scaffold (Docker Compose, Dockerfiles, FastAPI shell, Flutter + Next.js shells, migrations baseline, seed script skeleton, mailhog wired). Identity module (register/OTP/login) end-to-end.

**Week 4:** Wallet & Ledger module (double-entry, invariants enforced) + Payment module (Paystack sandbox integration, webhook, idempotency) end-to-end.

**Week 5:** Ticket module (skill question, free-entry transcription, idempotent issuance) + one active seed draw end-to-end.

**Week 6:** Draw Engine module (commit, sale, close, reveal with real Bitcoin + drand entropy, deterministic winner selection, hash-chained audit log) + Admin flows.

**Week 7:** Public proof page + trust-story pages + polish + seed-data reset script + demo-mode config (auto-advance draw timers so demo doesn't wait real minutes).

### Post-week-7: Demo prep

- Rehearsal: founder walks the full demo end-to-end 20+ times.
- Fixes on any friction discovered.
- Screen recording of the "hero flow" as a fallback for demos where live sharing fails.

### Parallel to the above

- Counsel engagement runs its own 4–6 week track (founder-owned).
- The V0.5 demo can be shown to counsel from Week 6 onwards.

## 6. Success gates

Demo is "ready to show investors" when all of these are true:

- [ ] Every flagship flow step (§2, all 16) runs end-to-end without error on a clean laptop.
- [ ] `docker compose up` starts everything in < 60 seconds.
- [ ] Demo reset (`make demo-reset` or equivalent) wipes and reseeds in < 30 seconds.
- [ ] Audit-log verifier script produces same winner when re-run against published proof.
- [ ] Design pass sign-off by founder.
- [ ] Founder walks demo 20+ times without a stumble.
- [ ] Fresh clone works on a second engineer's laptop in < 15 minutes.
- [ ] Screen-recording fallback exists.

## 8. Risks specific to V0.5

Full risk implications amended into `docs/risk-register.md`. Three worth calling out here:

1. **Demo becomes the design.** Shortcuts made for demo speed harden into V1 patterns because "it worked in the demo, why change it?" **Mitigation:** every demo shortcut is documented and ticketed for V1 removal at time of introduction. Ticketed items block Phase 3 exit.
2. **Investor demo underperforms.** Demo has bugs, tells wrong story, or fails to land the trust proof. **Mitigation:** 20+ rehearsals with founder before first external showing; screen-recording fallback; scope kept tight to increase reliability per feature.
3. **Time-slip on design pass.** Design work can expand. **Mitigation:** hard 2-week cap on design pass; if uncomfortable at week 2, Sally hands off "good enough" wireframes with a "V1 design refresh" ticket rather than delaying build start.
